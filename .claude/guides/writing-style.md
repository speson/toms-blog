# Tom's Blog Writing Style Guide

## 개요

**목표**: AI가 작성한 듯한 딱딱한 뉴스 스타일에서 **Tom의 개인적인 목소리**가 담긴 대화체로 전환

**참고 스타일**: https://goddaehee.tistory.com/504

---

## 핵심 변화

### ❌ 이전 방식 (DeveloperPerspective)

```markdown
## OpenCode란?

OpenCode는 터미널 기반 AI 코딩 에이전트입니다.

<DeveloperPerspective>
이 도구는 개발자에게 매우 유용합니다.
</DeveloperPerspective>
```

### ✅ 새로운 방식 (통합된 개인적 목소리)

```markdown
## OpenCode란? (그리고 제가 선택한 이유)

요즘 AI 코딩 도구가 많이 나오고 있는데, 그중에서도 OpenCode는 좀 특별한 존재입니다.

터미널에서 실행되는 오픈소스 AI 코딩 에이전트인데, **Claude Code와 비슷하지만 100% 오픈소스**라는 점이 가장 큰 매력입니다. Claude Code는 좋지만 결제해야 하잖아요? OpenCode는 무료로 쓸 수 있고, 원하는 AI 모델을 골라 쓸 수도 있습니다.

제가 실제로 써본 결과, LSP 지원이 기본 내장되어 있어서 코드 분석이 정말 정확합니다. 특히 TypeScript 프로젝트에서 타입 에러를 찾아내는 능력이 인상적이었어요.

💡 **제가 OpenCode를 선택한 이유:**

- Claude Code는 월 $20인데, OpenCode는 공짜
- Claude, GPT, Gemini 중에서 AI 모델을 골라 쓸 수 있음
- LSP 지원이 기본 내장되어 있어서 코드 분석이 정확함
```

---

## 핵심 원칙

### 1. **대화체 사용**

**❌ 피하기:**

- "~입니다" (너무 formal)
- "~한다" (명령조/객관적)
- 사실羅列만 하는 뉴스체

**✅ 사용하기:**

- "~예요/~이에요" (friendly)
- "~해요" (conversational)
- "~더라고요" (personal experience)
- "~겁니다" (assertive but warm)

### 2. **개인적 경험 녹이기**

**Before:**

```
OpenCode는 터미널 기반 AI 코딩 에이전트입니다.
```

**After:**

```
요즘 AI 코딩 도구가 많이 나오고 있는데, 그중에서도 OpenCode는 좀 특별한 존재입니다.

터미널에서 실행되는 오픈소스 AI 코딩 에이전트인데, Claude Code와 비슷하지만 100% 오픈소스라는 점이 가장 큰 매력입니다. Claude Code는 좋지만 결제해야 하잖아요? OpenCode는 무료로 쓸 수 있고, 원하는 AI 모델을 골라 쓸 수도 있습니다.

제가 실제로 써본 결과, LSP 지원이 기본 내장되어 있어서 코드 분석이 정말 정확합니다. 특히 TypeScript 프로젝트에서 타입 에러를 찾아내는 능력이 인상적이었어요.
```

### 3. **인라인 코멘터리 (이모지 활용)**

별도의 섹션 없이 자연스럽게 의견을 녹입니다:

- 💡 **팁/인사이트**: "여기서 중요한 점은..."
- 🎯 **강추**: "제가 가장 좋아하는 기능은..."
- ⚠️ **주의**: "주의할 점은..."
- 📝 **참고**: "팁: ..."
- 🤔 **개인적 생각**: "솔직히 말하면..."
- 💰 **비용 관련**: "비용 절약 팁: ..."

### 4. **구조 변화**

**Old Structure:**

1. Facts
2. Tables
3. Code blocks
4. DeveloperPerspective (separate section)

**New Structure:**

1. **Personal Hook** - 왜 이 주제가 중요한지 (개인적 연결)
2. **What it is** - 간단한 설명 + 내 경험
3. **My Experience** - 실제 테스트/사용 후기
4. **Key Insights** - 내가 발견한 특별한 점
5. **Practical Guide** - 실용적 가이드 + 팁
6. **Verdict** - 최종 의견/추천

## 작성 템플릿

### 헤더 (Frontmatter)

```yaml
---
title: "[제품명] 사용 후기: [한 줄 요약]"
description: "[개인적 경험을 담은 2-3문장]"
date: "YYYY-MM-DD"
tags: ["태그1", "태그2", "태그3"]
source: "original" # 또는 "geeknews", "openai" 등
sourceUrl: "https://..." # 원문 URL (있는 경우)
category: "ai-news" # ai-news, updates, opensource
draft: false
---
```

## 어조/말투 참고

### 자주 사용하는 표현

**시작:**
- "안녕하세요, Tom입니다."
- "이번에 소개할..."
- "요즘 [주제]가 많이 나오고 있는데..."

**의견 표현:**
- "~더라고요" (personal discovery)
- "~것 같아요" (opinion)
- "~이 인상적이었어요" (positive impression)
- "~가 아쉬웠어요" (negative but soft)

**추천:**
- "추천드립니다"
- "꼭 써보세요"
- "한 번 고려해보세요"

**경고/주의:**
- "주의할 점은..."
- "이 부분은 꼭 확인하세요"
- "~지 않도록 조심하세요"

**마무리:**
- "여러분도 한 번 써보세요."
- "생각보다 괜찮을 겁니다."
- "궁금한 점 있으시면 댓글 남겨주세요."

## 체크리스트

포스트 작성 후 확인:

- [ ] 인트로에 개인적 연결점이 있는가?
- [ ] "~입니다"보다 "~예요/해요"가 더 많은가?
- [ ] 개인적 경험/의견이 자연스럽게 녹아있는가?
- [ ] 이모지(💡🎯⚠️)를 활용했는가?
- [ ] 단순 사실羅列보다 "내 생각"이 담겨있는가?
- [ ] 마무리에 독자를 향한 인사/추천이 있는가?
- [ ] 따옴표+볼드는 `"**텍스트**"` 형태인가? (`**"텍스트"**` ❌)
